The Problem We're Solving

Navigation systems today are stuck in the past. Whether it's Google Maps, Tesla's Autopilot, or industrial robotics, they all share one fatal flaw: they optimize purely for speed. Get from A to B as fast as possible, consequences be damned.

But real-world travel isn't one-dimensional. A parent driving with kids values safety over shaving off two minutes. A delivery driver on a 12-hour shift needs comfort to avoid fatigue. A tourist wants scenic routes, not just efficient ones. Current systems ignore these human needs entirely.

Our Solution: Context-Aware, Multi-Criteria Routing

Vehicle AutoRoute Navigator introduces a paradigm shift. Instead of blindly chasing the fastest route, we evaluate paths across multiple dimensions‚Äîtime, safety, comfort, and scenic value‚Äîthen dynamically adapt as conditions change. When our robot encounters unexpected obstacles, rough terrain, or hazardous conditions, it doesn't just stop or fail‚Äîit intelligently re-routes to a better path.

This isn't theoretical. We're demonstrating it live on a TurtleBot3, using only onboard sensors. No external cameras, no cloud processing‚Äîjust smart algorithms running at the edge.

The Math Behind the Magic

Our routing algorithm uses a modified A* search that scores each candidate path with a weighted cost function:

Cost = wtime √ó Time + wrisk √ó Risk + wcomfort √ó Comfort ‚àí wscenic √ó Scenic

Breaking it down:

  • Time: Traditional routing distance/speed calculation
  • Risk: Obstacles detected by LIDAR, simulated weather hazards, and historical incident data
  • Comfort: Computed from IMU vertical acceleration variance (rough roads show high variance)
  • Scenic: Pre-tagged routes or areas marked as visually appealing (subtracted because it's a benefit)

The weights change based on user profile. A "Fastest" profile weights time heavily while a "Safest" profile prioritizes minimizing risk. Users can blend profiles on the fly with a simple slider.

System Architecture

We've designed this as a distributed system with clear separation of concerns. Each component can be upgraded or replaced without breaking the whole stack‚Äîcritical for production robotics systems.

Router Module

Python-based A* implementation. Builds candidate paths, applies cost function, re-ranks on condition changes.

Signals Node

ROS 2 node that aggregates LIDAR obstacles, IMU roughness metrics, and external context signals.

Waypoint Client

Interfaces with Nav2's FollowWaypoints action server. Monitors progress, cancels/reissues when re-planning triggers.

Web Gateway

FastAPI server with WebSocket support. Streams live robot pose to the UI, accepts destination commands.

Multi-Criteria Route Selection

Same origin and destination, three different optimal paths. Each profile weights the cost function differently, resulting in fundamentally different route choices.

Dynamic Re-Planning in Action

Real-time adaptation when conditions change. Our system detects obstacles via LIDAR, evaluates alternative routes, and executes a new path‚Äîall in under 2 seconds with zero human intervention.

Explainable User Interface

Transparency is key to trust. Our web interface doesn't just show you where the robot is going‚Äîit explains WHY it chose that route. Every decision is backed by quantified trade-offs.
